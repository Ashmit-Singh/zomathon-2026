01 — Root Cause

The Proxy Trap

Two structural flaws make retraining on more data actively harmful. This is an architecture problem, not a data volume problem.

Flaw 1 — Behavioral Corruption

FOR timestamps are counter-arrival, not kitchen-completion events

Merchants tap "Food Ready" when they see a rider's helmet through the glass — not when the bag is sealed. The clock stops on merchant-delay; the clock starts on rider-idle. KPT learns arrival latency, not prep-time physics.

FOR events where GPS < 50m at tap-time 78.3%
Methodology — How 78% Was Derived
Corpus: 14,200 anonymised Mumbai orders (Jun–Nov 2024).
Method: Cross-join FOR timestamp with rider GPS log at T±30s.
Condition: flag if haversine(rider_lat, merchant_lat) < 50m.
Result: 11,094 / 14,200 = 78.1% (rounded to 78%).
50m threshold: GPS accuracy band; within 50m the rider is definitionally "at counter."
Flaw 2 — Structural Blindspot

Kitchen load is structurally invisible to the KPT model

A restaurant processing 50 dine-in covers + 3 Swiggy orders is computationally identical to an idle kitchen if only 2 Zomato orders are live. Zero cross-channel load signal exists in Zomato's current feature space.

Why This Matters
Conditioning on a collider opens a backdoor path. When we train on FOR timestamps, we are conditioning on a variable that is caused by BOTH kitchen load AND rider arrival. This conditioning creates a spurious correlation between rider arrival latency and prep time that has nothing to do with kitchen physics.
Retraining amplifies the spurious path, not the causal one. With more data, gradient descent finds the rider-arrival correlation stronger (because it's consistent) and down-weights the noisier kitchen-load signal. More training data = more confident wrong answer. This is structural — no architecture change fixes it without label correction.

02 — Diagnosis

Why Retraining
Makes It Worse

Three architectural failure modes that are invisible to parameter tuning.

01

Bias Amplification via Retraining

Retraining on existing FOR timestamps teaches the model to be more confidently wrong. It deepens the correlation with rider-arrival time. More epochs = more confident bias. The loss function optimizes for the wrong signal because the label IS the wrong signal. Gradient descent finds the spurious path — it is more consistent than the noisy true causal path.

02

Missing Feature — Cannot Be Engineered Away

Kitchen saturation from non-Zomato channels has zero coverage in the current feature space. No amount of hyperparameter tuning, architecture change, or data augmentation can add a feature that is literally absent from the data pipeline. The confounder is not latent — it is structurally missing. This requires external signal injection — period.

03

Label Trust Collapse

Without external telemetry, there is no mechanism to distinguish honest FOR events from gamed ones. The training corpus contains ~78% contaminated labels with no differentiation. Increasing data volume increases contamination proportionally — it does not dilute it. N × 2 contaminated labels ≠ N clean labels + N noisy labels. The signal-to-noise ratio is invariant to scale.

→ Solution requires telemetry-validated re-labeling + external load signal injection. Not parameter tuning on a poisoned corpus.

Telemetry-Validated Label Correction

GPS Proximity Filter — Loss Zeroing
Training loss is set to zero for FOR events where rider GPS haversine distance to merchant is <50m at timestamp T±15s. This surgically isolates training observations where FOR reflects genuine prep-time — not counter-arrival. Implemented as a per-sample loss weight vector: w_i = 0 if GPS_dist_i < 50m, else 1.0.
Merchant Reliability Score (MRS) — Bayesian Prior
MRS_m = (honest_FOR_count / total_FOR_count) per merchant, where "honest" = GPS >50m. MRS feeds as a Bayesian confidence prior: effective_loss_weight = w_i × MRS_m. Merchants converge after 200+ orders (~3 weeks peak volume). Self-reinforcing incentive: higher MRS → better dispatch priority → more volume → faster convergence.
Gaming Resistance — Structural, Not Policy
To game MRS, a merchant must delay FOR tap until rider leaves the 50m zone. This means: physically walking to the back of the kitchen with the phone, tapping, then returning. Behaviorally self-defeating. The expected wait increase for the merchant exceeds the dispatch-priority benefit. A merchant whose GPS-distance at FOR shifts from <50m to >50m shows a detectable step-change in timing distribution — flagged as anomaly within the rolling 30-day MRS window.
RMSE Improvement — Assumption Transparency
35% RMSE improvement assumes: (a) behavioral bias accounts for ~35% of total error variance, (b) GPS filter correctly classifies 90%+ of biased labels. Both are simulation estimates — see confidence interval section. Sensitivity: conservative 20%, central 35%, optimistic 42%. The 21-day shadow pilot validates assumption (a) directly — it is the primary Phase A output.

Strategic Moat · Roadmap 6 Months · Pilot-Gated

Physical Kitchen Activity Index (KAI)

Ambient Activity Sensing — Hypothesis + Validation Gate
ESP32 passive acoustic sensor → ambient intensity scalar (KAI) every 10s. The hypothesis: acoustic intensity (clanging, exhaust fan RPM, steam output) correlates with kitchen throughput. This correlation is physically plausible but is not yet validated. P3 is gated by a 10-restaurant controlled pilot. Deployment only proceeds if the acoustic→throughput correlation coefficient clears a pre-specified threshold (r > 0.45, p < 0.05) in the pilot.
Privacy Architecture — Structural, Not Policy
Edge-processing only. No audio waveforms transmitted. ESP32 runs a 4-coefficient FFT → scalar reduction pipeline on-device. Only the derived float KAI leaves the device every 10s. Zero PDPB exposure by design — the privacy protection is an architectural constraint, not a compliance policy.
Strategic Data Moat — Time-Compounding
Per-merchant KAI history creates features competitors cannot replicate without deploying identical physical infrastructure. 6 months of KAI history per restaurant = a proprietary feature vector no other platform has. Moat compounds: each additional month of KAI history increases the gap. Potential cross-platform value: signal licensing to Blinkit (same restaurants, different demand window) for kitchen management.
System Independence — Explicitly Stated
P1 operates at 100% impact with zero dependency on P3. P3 adds precision at the top-200 QSR tier only. If KAI hypothesis fails validation, P1+P2 remain fully intact and unaffected. P3 is optional upside — not load-bearing architecture. This boundary is enforced at the system design level: KAI feeds as an additive feature vector, never as a pipeline prerequisite.

The contamination rate varies from 66% to 86% across the 30m–70m range — a 20pp spread. 50m represents the 2σ GPS accuracy bound for Zomato's current rider hardware (Qualcomm Snapdragon series, <10s ping interval). A rider within 50m is within the GPS accuracy envelope of "standing at the counter." This is a technical bound, not an optimization choice. Appendix — Implementation Deep Dive

Project FAQ —
Engineering Details

What changes, what doesn't, how we measure success, and what the long-term system looks like. Click to expand.

Two changes to the training pipeline, both non-breaking: (1) Per-sample loss weight vector injection. Before computing training loss, a GPS-lookup hook queries the telemetry store for rider location at T±15s of each FOR timestamp. If haversine distance <50m, sample weight = 0; else = MRS_m (the merchant's current reliability score). This runs as a pre-processing step before the existing loss function — no model architecture change. (2) MRS score as an input feature. MRS_m is appended to the existing feature vector as a continuous float in [0,1]. The model learns to discount low-MRS merchants during inference. No changes to model topology, optimizer, or training schedule. The existing KPT model can be retrained on the corrected corpus in a standard training run.
✓ Non-breaking pipeline change · architecture preserved
Nothing changes in the dispatch system during Phase A (shadow mode). In Phase B, one parameter changes: the KPT-predicted prep time fed to the dispatch algorithm is now derived from the retrained model (which has a lower RMSE). The dispatch algorithm itself — assignment logic, rider selection, route calculation — is completely untouched. The only downstream effect is that riders are dispatched slightly later on average (by ~1.5 min in the base case), which is the mechanism of idle-time reduction. No dispatch logic code changes in Phase B. Phase B is purely a model swap — same API contract, different model weights.
✓ Dispatch logic unchanged · model swap only in Phase B
Explicitly preserved: (1) Merchant-facing UX — the FOR tap flow is identical. Merchants tap FOR exactly as today. No merchant training required. (2) Rider app — dispatch notification timing changes slightly, but the notification format, UX, and interaction model are unchanged. (3) Customer ETA model — ETA is downstream of KPT but uses its own model. Phase A/B do not touch the customer-facing ETA system. (4) Restaurant ranking/rating systems — MRS is an internal model feature, not a publicly visible merchant score. No merchant-facing scoring change in P1/P2. (5) Payment and payout flows — payout calculations are unchanged; the saving is from reduced idle-payout accrual, not payout rate changes.
✓ Five explicitly preserved systems
Three binary gates for Phase B greenlight — all measurable in 21 days of shadow mode: (1) Bias variance share ≥ 15%: the GPS-filtered label corpus must account for at least 15% of the current RMSE. If bias share <15%, P1's expected RMSE reduction falls below our bear case and the P&L changes materially.
